# Groundwork + Spec-Driven Development Integration

## The Complementary Relationship

Spec-Driven Development (SDD) and Groundwork solve different parts of the same problem:

| **Aspect** | **SDD Tools** | **Groundwork** |
|------------|---------------|----------------|
| **Timing** | Before building | During building |
| **Focus** | What to build | What was decided |
| **Maintenance** | Manual | Automatic |
| **Enforcement** | Guidance only | Hard blocks |
| **Source of truth** | Static documents | Living graph |

## Combined Workflow

```
┌─────────────────────────────────────────────────────────┐
│  BEFORE BUILDING (SDD)                                   │
│  • Write specification using SDD tools                   │
│  • Define features, acceptance criteria, constraints     │
│  • Load spec into AI tools as starting context           │
└────────────────────┬────────────────────────────────────┘
                     │
                     ▼
┌─────────────────────────────────────────────────────────┐
│  DURING BUILDING (Groundwork)                            │
│  • AI executes against spec                              │
│  • Mid-build decisions captured automatically            │
│  • Decisions propagated to all team members              │
│  • Conflicts detected in real-time                       │
└────────────────────┬────────────────────────────────────┘
                     │
                     ▼
┌─────────────────────────────────────────────────────────┐
│  AFTER BUILDING (Groundwork → SDD)                       │
│  • Captured decisions fed back into spec                 │
│  • Spec stays current without manual work                │
│  • PR checked against decision graph                     │
│  • Next feature starts from updated foundation           │
└─────────────────────────────────────────────────────────┘
```

## Integration With Major SDD Tools

### GitHub Spec Kit

**What it does:**
- 90,000+ GitHub stars, MIT licensed
- Four-stage workflow: Specification → Plan → Tasks → Implementation
- Works with Claude Code, Copilot, Cursor, Gemini
- Model-agnostic

**Where Groundwork fits:**
```typescript
// .github/spec-kit/project.md (written by developer using Spec Kit)
# Authentication System

## Specification
- JWT-based authentication
- User sessions expire after 7 days
- Refresh tokens stored in HTTP-only cookies

## Plan
1. Implement JWT signing/verification
2. Create session middleware
3. Build refresh token rotation

## Tasks
- [ ] Install jsonwebtoken library
- [ ] Create auth service
- [ ] Add session middleware
```

During implementation, developers make additional decisions:
- Which specific JWT algorithm (RS256 vs HS256)
- Where to store JWT secrets
- How to handle token expiration edge cases

**Groundwork captures these automatically** and feeds them back:

```typescript
// Groundwork auto-updates the spec file:
# Authentication System

## Specification
... (original content) ...

## Implementation Decisions (auto-generated by Groundwork)
- **Decision #47**: Using RS256 for JWT signing (made by @sarah, Jan 15)
  - Rationale: Need key rotation without redeploying services
  - Alternative considered: HS256 (simpler but requires shared secret)
  
- **Decision #48**: JWT secrets stored in AWS Secrets Manager (made by @mike, Jan 15)
  - Rationale: Rotation support, audit trail
  - Affects: Auth service, API gateway
```

**Integration method:**
- Groundwork adds a file watcher for `spec-kit/` directories
- When decisions are extracted, appends them to relevant spec files
- Uses GitHub App to commit updates automatically
- Maintains a separate section to avoid conflicts with human edits

### OpenSpec

**What it does:**
- Lightweight SDD framework (open source)
- Two-folder model: `openspec/specs/` (current) and `openspec/changes/` (proposed)
- Each change has: proposal, design, and tasks document
- Brownfield-first design (works with existing codebases)
- Built-in ADR (Architectural Decision Records) support

**Where Groundwork fits:**

OpenSpec already has a schema for ADRs published in April 2026:

```markdown
# openspec/specs/auth-system/decisions.md

## ADR-001: Use JWT for Authentication
**Status**: Accepted
**Date**: 2026-01-10
**Author**: @sarah

### Context
Need stateless authentication for microservices architecture.

### Decision
Implement JWT-based authentication with RS256 signing.

### Consequences
- No shared session state required
- Need key rotation strategy
- Slightly larger token size than sessions
```

**Problem**: These ADRs must be written manually. Nobody maintains them consistently.

**Groundwork solution**: Auto-populate ADR sections from extracted decisions.

**Technical integration:**
```typescript
// Groundwork OpenSpec Plugin
export class OpenSpecPlugin {
  async onDecisionExtracted(decision: Decision) {
    // Find relevant OpenSpec spec directory
    const specDir = await findRelevantSpec(decision.affectedModules);
    
    if (!specDir) return;
    
    // Check if ADR file exists
    const adrPath = join(specDir, 'decisions.md');
    const adrContent = await readFile(adrPath).catch(() => '');
    
    // Generate ADR entry
    const adrEntry = formatAsADR(decision);
    
    // Append to file
    await appendFile(adrPath, '\n\n' + adrEntry);
    
    // Commit via OpenSpec tooling
    await openspec.commit({
      message: `Add ADR from Groundwork decision #${decision.id}`,
      files: [adrPath]
    });
  }
}
```

**Benefits:**
- OpenSpec users get ADRs automatically maintained
- Groundwork benefits from OpenSpec's governance workflow
- ADRs stay in sync with actual implementation

### AWS Kiro

**What it does:**
- Standalone IDE built specifically for SDD
- Spec, plan, tasks, and code all in one workspace
- Deep AWS integration
- Commercial product
- Customer case study: 40-hour features in < 8 hours

**Where Groundwork fits:**

Kiro is an IDE. Groundwork is infrastructure that runs beneath IDEs.

**Integration approach:**
- Kiro users install Groundwork MCP server
- Decisions made in Kiro sessions get captured by Groundwork
- When multiple developers use Kiro on same project, Groundwork shares decisions between them
- Kiro's within-session spec discipline + Groundwork's cross-session memory = complete solution

**Technical integration:**
```typescript
// Kiro has MCP support built-in
// Groundwork works automatically once MCP server is running

// Kiro-specific enhancement: Spec file sync
export class KiroPlugin {
  async syncToKiroSpec(decision: Decision) {
    // Kiro stores specs in .kiro/specs/
    const kiroSpecPath = '.kiro/specs/architectural-decisions.json';
    const spec = await readJSON(kiroSpecPath);
    
    spec.decisions.push({
      id: decision.id,
      title: decision.title,
      domain: decision.domain,
      timestamp: decision.madeAt,
      source: 'groundwork-auto-captured'
    });
    
    await writeJSON(kiroSpecPath, spec);
  }
}
```

### BMAD-METHOD

**What it does:**
- 48,400+ GitHub stars, open source
- Orchestrates multiple specialized AI agents with named personas:
  - Mary (Business Analyst)
  - Preston (Product Manager)
  - Winston (Architect)
  - Devon (Developer)
  - Quinn (QA)
- Role-based structure with strict responsibilities

**Where Groundwork fits:**

BMAD's biggest gap: **cross-session memory between personas**.

Scenario without Groundwork:
1. Winston (Architect) decides to use PostgreSQL in Monday's session
2. Session ends
3. Devon (Developer) starts fresh session on Tuesday
4. Devon's AI has no memory of Winston's decision
5. Devon suggests MongoDB (different technology)
6. Conflict!

**Groundwork solution:**
```typescript
// When Winston makes architectural decision
Session: Winston (Architect)
Winston: "We'll use PostgreSQL with Prisma ORM"
↓
Groundwork extracts: 
  Decision #12: PostgreSQL as primary database (P0)
  Decision #13: Prisma ORM for data access (P0)
↓
Stored in graph with metadata: { persona: 'Winston', role: 'Architect' }

// Next day, Devon starts
Session: Devon (Developer)
↓
Groundwork injects before Devon's first response:
  "The following architectural decisions have been made:
   - #12: PostgreSQL (decided by Winston, Architect)
   - #13: Prisma ORM (decided by Winston, Architect)"
↓
Devon's AI now knows the constraints from Winston's session
```

**BMAD-specific enhancement:**

Respect persona hierarchy:
```typescript
const personaHierarchy = {
  'Winston': { role: 'Architect', priority: 1 },
  'Preston': { role: 'Product Manager', priority: 2 },
  'Devon': { role: 'Developer', priority: 3 },
  'Quinn': { role: 'QA', priority: 4 }
};

// When injecting decisions into Devon's session:
// - Winston's decisions are marked as "architectural constraints" (cannot override)
// - Preston's decisions are marked as "product requirements" (cannot override)
// - Devon's own decisions are marked as "implementation details" (can iterate)
```

### Tessl

**What it does:**
- Most aggressive SDD approach
- Pushes toward "spec-as-source" — humans only author specs, all code generated
- Cisco case study: 47% baseline → 84% accuracy with Tessl
- Commercial product

**Where Groundwork fits:**

Tessl's own blog acknowledges:
> "The same spec will produce different code from different agents."

**Problem**: Spec is deterministic. AI execution is not.

Different developers' agents might:
- Choose different libraries for same requirement
- Structure code differently
- Make different trade-offs

**Groundwork solution**: Ensure consistency across agent executions.

```
Spec (written once):
  "Build authentication system with JWT"

Developer 1's agent executes:
  ↓ chooses jsonwebtoken library
  ↓ chooses RS256 algorithm
  ↓ Groundwork captures these decisions

Developer 2's agent executes:
  ↓ Groundwork injects: jsonwebtoken + RS256 already decided
  ↓ Agent uses same choices
  ✓ Consistency maintained
```

**Integration approach:**
- Tessl generates code from specs
- Groundwork captures how the code was generated
- Next execution uses prior generation decisions
- Consistency across team, even from identical specs

## Generic Integration Pattern

For any SDD tool, Groundwork can integrate via three methods:

### Method 1: File Watcher Integration
```typescript
// Watch spec files for changes
chokidar.watch('specs/**/*.md').on('change', async (path) => {
  // Extract any new decisions from spec
  const decisions = await extractFromSpecFile(path);
  await groundwork.addDecisions(decisions, { source: 'spec-file' });
});

// Watch for new Groundwork decisions
groundwork.on('decision:extracted', async (decision) => {
  // Append to relevant spec file
  await appendToSpecFile(decision);
});
```

### Method 2: MCP Protocol (Universal)
```typescript
// All MCP-compatible tools work automatically
// No custom integration needed
// Groundwork MCP server handles everything
```

### Method 3: API Integration
```typescript
// SDD tool calls Groundwork API
const groundwork = new GroundworkClient(apiKey);

// Before generating code
const decisions = await groundwork.getRelevantDecisions({
  module: 'auth',
  domain: 'Security'
});

// After completing task
await groundwork.recordDecision({
  title: 'Using bcrypt for password hashing',
  domain: 'Security',
  priority: 'P0',
  source: 'sdd-tool-xyz'
});
```

## Technical Integration Guides

### OpenSpec Integration (Detailed)

**Step 1: Install Groundwork**
```bash
npm install -g @groundwork/cli
cd your-openspec-project
groundwork init
```

**Step 2: Enable OpenSpec plugin**
```bash
groundwork plugin install openspec
```

**Step 3: Configure**
```yaml
# .groundwork/config.yml
plugins:
  openspec:
    enabled: true
    adr_path: openspec/specs/{module}/decisions.md
    auto_commit: true
    commit_message: "chore: update ADRs from Groundwork"
```

**Step 4: Work normally**
- Write OpenSpec specs as usual
- Build features with AI tools
- Groundwork captures decisions
- ADR files auto-update
- Commit with your regular workflow

### GitHub Spec Kit Integration (Detailed)

**Step 1: Add Groundwork to your Spec Kit project**
```bash
cd your-spec-kit-project
groundwork init --integration=spec-kit
```

**Step 2: Groundwork creates new section in spec files**
```markdown
# Feature Specification

## Specification
[Your original content]

## Plan
[Your original content]

## Tasks
[Your original content]

## Implementation Decisions (Groundwork)
<!-- AUTO-GENERATED - DO NOT EDIT MANUALLY -->
<!-- This section is maintained by Groundwork -->

- **#23**: Using Redis for session storage
  - Made by @sarah on 2026-01-15
  - Rationale: Need distributed sessions for horizontal scaling
  - Affects: auth-service, api-gateway

<!-- END GROUNDWORK SECTION -->
```

**Step 3: Enable auto-sync**
```yaml
# .github/workflows/groundwork-sync.yml
name: Sync Groundwork Decisions

on:
  schedule:
    - cron: '0 * * * *' # Every hour

jobs:
  sync:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - uses: groundwork/sync-action@v1
        with:
          project-id: ${{ secrets.GROUNDWORK_PROJECT_ID }}
          spec-kit-path: .github/spec-kit/
```
